1. System Architecture Overview

PropCRM follows a three-tier architecture with clear separation between the presentation layer (GitHub Pages), application layer (Google Apps Script), and data layer (Google Sheets). All communication between tiers occurs over HTTPS using JSON payloads.

1.1 Architecture Layers

LayerComponentTechnologyResponsibility
PresentationLead FormHTML/CSS/JS (GitHub Pages)Lead data capture, client-side validation, UX feedback
PresentationAgent DashboardHTML/CSS/JS + Chart.js (GitHub Pages)Lead management, analytics visualization, email actions
ApplicationAPI GatewayGoogle Apps Script (Web App)Request routing, server-side validation, business logic, email orchestration
DataLead StoreGoogle Sheets — "Leads" sheetPersistent storage of all lead records
DataConfig StoreGoogle Sheets — "AgentConfig" sheetAgent settings, API keys, authentication PIN
ExternalEmail ServiceBrevo v3 REST APITransactional email delivery

4. Data Flow Scenarios

4.1 Flow 1: Lead Submission (Happy Path)

Trigger: Customer clicks "Submit" on the lead form

1
Client-Side Validation Browser validates all fields (name length, email format, area non-empty, price > 0). If invalid → show inline errors, halt submission.
2
HTTP POST to Apps Script Form data sent as JSON via fetch() to the Apps Script Web App URL. Submit button shows loading spinner.
3
Server-Side Validation Apps Script re-validates all fields server-side. If invalid → returns { success: false, errors: [...] }.
4
Generate Lead Record System generates UUID, calculates price bracket, captures timestamp. Constructs complete lead record.
5
Write to Google Sheets Appends new row to the "Leads" sheet with all 11 columns. Default values: contacted = FALSE, emailSentToLead = FALSE, reminderSentToAgent = FALSE.
6
Send Acknowledgment Email to Lead Apps Script reads Brevo API key from AgentConfig, sends acknowledgment email to the lead's email via Brevo v3 API. Updates emailSentToLead = TRUE.
7
Send Notification Email to Agent Apps Script sends a notification email to the agent's email (from AgentConfig) with the lead's details.
8
Return Success Response Apps Script returns { success: true, id: "..." }. Frontend shows success toast, resets form.

4.2 Flow 2: Dashboard Data Load

Trigger: Agent opens dashboard and enters correct PIN

1
PIN Verification Agent enters PIN → POST to Apps Script with action: "verifyPin". On success, PIN is stored in session memory (JavaScript variable). On failure, error message shown.
2
Fetch Leads Data GET request to Apps Script with action=getLeads&pin=XXXX. Apps Script verifies PIN, reads all rows from "Leads" sheet.
3
Compute Statistics (Server-Side) Apps Script computes: total leads, contacted count, uncontacted count, contacted %, average price, price bracket distribution (count per bracket), area distribution (count per locality).
4
Return Aggregated Response Returns JSON with leads[], stats{}, priceBrackets{}, areaCounts{}.
5
Render Dashboard Frontend populates: stat cards, Chart.js doughnut chart (price brackets), Chart.js bar chart (area distribution), leads data table. Green gradient colors applied based on data concentration.

4.3 Flow 3: Mark Lead as Contacted

Trigger: Agent checks the "Contacted" checkbox on a lead row

1
Checkbox Toggle Event JavaScript captures change event on the checkbox for the specific lead row.
2
POST Update Request Sends { action: "updateContacted", id: "...", contacted: true, pin: "..." } to Apps Script.
3
Update Google Sheet Apps Script finds the row by ID, sets contacted = TRUE and contactedAt = NOW().
4
UI Confirmation Frontend updates the row visually (e.g., reduced opacity or green highlight) and updates stat cards.

4.4 Flow 4: Send Follow-up Email from Dashboard

Trigger: Agent clicks "Send Email" button on a lead row

1
Button Click Event Agent clicks "Send Email" → button changes to loading state.
2
POST Email Request Sends { action: "sendEmailToLead", id: "...", pin: "..." } to Apps Script.
3
Apps Script Processes Email Reads lead data by ID from Sheets, reads agent config (name, Brevo key), constructs follow-up email HTML, sends via Brevo API.
4
UI Confirmation Button shows ✓ checkmark with "Sent" text. Reverts to normal after 3 seconds.

4.5 Flow 5: Automated Reminder for Uncontacted Leads

Trigger: Time-driven trigger (runs every 1 hour automatically in Apps Script)

1
Scheduled Trigger Fires Google Apps Script time-driven trigger executes checkUncontactedLeads() every 1 hour.
2
Query Uncontacted Leads Reads "Leads" sheet. Filters for rows where: contacted = FALSE AND reminderSentToAgent = FALSE AND submittedAt is more than reminderHours ago.
3
Send Reminder Emails For each qualifying lead, sends a reminder email to the agent via Brevo with lead details and urgency message.
4
Update Reminder Flag Sets reminderSentToAgent = TRUE for each processed lead to prevent duplicate reminders.

5. Data Security & Privacy

5.1 Sensitive Data Handling

Data ItemClassificationStorageProtection
Brevo API KeySecretAgentConfig sheet (cell C1)Never sent to frontend; accessed only by Apps Script server-side
Dashboard PINSecretAgentConfig sheet (cell D1)Verified server-side; not stored in browser localStorage
Lead EmailPIILeads sheetAccessible only via authenticated dashboard (PIN required)
Lead NamePIILeads sheetSame as above

5.2 Security Controls

ControlImplementation
Transport encryptionAll communication over HTTPS (enforced by GitHub Pages and Apps Script)
API key protectionBrevo API key stored in Sheets, accessed only via Apps Script backend
Input sanitizationHTML entities escaped, script tags stripped before writing to Sheets
AuthenticationPIN verified on every dashboard API call (passed in request body/query)
Rate limitingInherent via Google Apps Script execution quotas (30 concurrent)
⚠ Security Limitation (v1): PIN-based authentication is suitable for single-agent use but is not enterprise-grade. The PIN is sent in request bodies over HTTPS, which is encrypted in transit but could be observed in browser developer tools. For multi-agent or sensitive deployments, upgrading to Google OAuth is recommended in v2.

6. Error Handling Strategy

Error ScenarioDetectionResponseUser Impact
Client-side validation failure Before API call Inline error messages on form fields Form not submitted; user corrects errors
Server-side validation failure Apps Script validation Returns { success: false, errors: [...] } Error toast shown; form data preserved
Google Sheets write failure Try-catch in Apps Script Returns { success: false, error: "Storage error" } Error toast; user asked to retry
Brevo API failure (email not sent) HTTP response status check Lead still saved; email flag stays FALSE; logged for retry Lead captured (no data loss); email delivery delayed
Network failure (frontend → Apps Script) Fetch catch block Error toast: "Network error. Please check your connection." Form data preserved; user retries
Invalid PIN on dashboard PIN verification endpoint Returns { success: false } Error message; login form remains visible
Apps Script quota exceeded Google error response HTTP 429 or error message Temporary unavailability; retry after cooldown
Design Principle: Email failures are non-blocking. The lead is always saved first. Email sending is a best-effort secondary action. If Brevo fails, the lead data is still safely stored in Google Sheets and visible on the dashboard.

7. Data Transformations

7.1 Price Bracket Calculation

Performed server-side in Apps Script when a new lead is submitted:

Input Price (₹)Bracket Output
0 – 24,99,999"Under 25L"
25,00,000 – 49,99,999"25L – 50L"
50,00,000 – 99,99,999"50L – 1Cr"
1,00,00,000 – 1,99,99,999"1Cr – 2Cr"
2,00,00,000 – 4,99,99,999"2Cr – 5Cr"
≥ 5,00,00,000"5Cr+"

7.2 Statistics Aggregation

Computed in Apps Script on every dashboard data fetch (GET request):

StatComputationOutput Format
Total LeadsCOUNT(all rows)Integer
Contacted CountCOUNT(WHERE contacted = TRUE)Integer
Uncontacted CountTotal - ContactedInteger
Contacted %(Contacted / Total) × 100Percentage (1 decimal)
Average PriceSUM(price) / COUNT₹ formatted
Price DistributionGROUP BY priceBracket → COUNTObject: { bracket: count }
Area DistributionGROUP BY area (case-insensitive) → COUNTObject: { area: count }, sorted desc
