1. Executive Summary

PropCRM is a lightweight, web-based Customer Relationship Management system designed specifically for independent real estate agents operating in the Indian market. The system captures property inquiry leads through a public-facing form, stores them in Google Sheets, and provides a secure agent dashboard for lead management, communication, and analytics.

The product eliminates the need for expensive CRM subscriptions by leveraging free-tier services — Google Sheets for data storage, Google Apps Script for backend logic, GitHub Pages for hosting, and Brevo for transactional email delivery.

Key Value Proposition: A zero-cost, fully functional CRM that an individual real estate agent can deploy in under 30 minutes, with automated lead notifications, email follow-ups, and visual analytics — without any recurring subscription fees.

2. Problem Statement

Independent real estate agents in India face several challenges in managing their leads:

  • Manual lead tracking: Leads arrive via phone calls, WhatsApp messages, website forms, and social media — scattered across platforms with no unified view.
  • No follow-up system: Without automated reminders, agents frequently miss or delay following up on leads, resulting in lost conversions.
  • No data visibility: Agents lack insight into which price brackets or localities generate the most interest, preventing them from optimizing their marketing spend and listings.
  • High CRM costs: Enterprise CRM tools (Salesforce, HubSpot, Zoho) are priced for teams and are overkill for a single agent handling 50–500 leads per month.

6. Functional Requirements

FR-1: Lead Capture Form

Req IDRequirementPriorityAcceptance Criteria
FR-1.1 Form shall contain fields: Full Name, Email Address, Property Area (locality), Property Price (₹) Must Have All four fields render correctly on desktop and mobile
FR-1.2 All fields are mandatory. Form shall not submit with any empty field. Must Have Submit button is disabled or shows error for incomplete forms
FR-1.3 Name must be at least 2 characters, alphabetic with spaces allowed Must Have Inline error message shown for invalid name
FR-1.4 Email must be a valid email format (RFC 5322 compliant regex) Must Have Inline error message shown for invalid email
FR-1.5 Price must be a positive number greater than zero Must Have Non-numeric or negative values rejected with error
FR-1.6 Form shall display a loading spinner during submission Should Have Submit button shows spinner, prevents double-click
FR-1.7 On successful submission, form shall display a success toast and reset Must Have Success message visible for 5 seconds, form fields cleared
FR-1.8 On failed submission, form shall display an error toast with retry option Must Have Error message visible, form data preserved for retry
FR-1.9 Form design shall use glassmorphism effect with cream and green color palette Should Have Visual review approved by stakeholder

FR-2: Data Storage (Google Sheets)

Req IDRequirementPriorityAcceptance Criteria
FR-2.1 Each lead shall be stored as a row in the "Leads" sheet with columns: ID, Name, Email, Area, Price, Price Bracket, Submitted At, Contacted, Contacted At, Email Sent To Lead, Reminder Sent To Agent Must Have New row appears in Sheets within 3 seconds of form submission
FR-2.2 Lead ID shall be a unique auto-generated identifier (UUID v4 or timestamp-based) Must Have No duplicate IDs across all leads
FR-2.3 Price bracket shall be auto-calculated: Under 25L, 25L–50L, 50L–1Cr, 1Cr–2Cr, 2Cr–5Cr, 5Cr+ Must Have Correct bracket assigned for boundary values (e.g., ₹25,00,000 → "25L–50L")
FR-2.4 Agent configuration shall be stored in a separate "AgentConfig" sheet Must Have Agent name, email, Brevo key, PIN, and reminder hours readable by the system
FR-2.5 Server-side validation shall reject malformed data before writing to Sheets Must Have Invalid submissions return HTTP 400 with error description

FR-3: Agent Dashboard

Req IDRequirementPriorityAcceptance Criteria
FR-3.1 Dashboard shall be protected by a PIN/password login screen Must Have Dashboard content hidden until correct PIN entered
FR-3.2 Dashboard shall display summary stats: Total Leads, Contacted %, Uncontacted Count, Average Property Price Must Have Stats update on page load and on manual refresh
FR-3.3 Price distribution shall be visualized as a doughnut/bar chart with percentage labels Must Have Chart segments match the 6 defined price brackets with green gradient coloring
FR-3.4 Area/locality distribution shall be visualized as a horizontal bar chart Must Have Top N localities displayed with lead count
FR-3.5 Chart colors shall follow green gradient: higher concentration → darker green Should Have Visual review confirms gradient is clear and readable
FR-3.6 Leads table shall display all lead data with columns: Name, Email, Area, Price, Date, Contacted, Actions Must Have Table is scrollable, sortable by date, and matches Sheets data
FR-3.7 Each row shall have a "Contacted" checkbox that updates the lead status in real-time Must Have Checkbox toggle updates Google Sheet within 3 seconds
FR-3.8 Each row shall have a "Send Email" button to send a follow-up email to the lead Must Have Email sent via Brevo, confirmation shown to agent
FR-3.9 Dashboard shall have a manual "Refresh" button to re-fetch latest data Should Have Data refreshes without full page reload

FR-4: Email System (via Brevo API)

Req IDRequirementPriorityTriggerAcceptance Criteria
FR-4.1 Send acknowledgment email to lead upon form submission Must Have Form submit → Apps Script Lead receives email within 60 seconds of submission
FR-4.2 Send new lead notification email to agent upon form submission Must Have Form submit → Apps Script Agent receives email with lead details within 60 seconds
FR-4.3 Send reminder email to agent for leads uncontacted beyond configured hours Must Have Scheduled trigger (every 1 hour) Reminder sent only once per lead; "reminderSent" flag set to TRUE
FR-4.4 Agent can send a follow-up email to any lead from the dashboard Must Have Agent clicks "Send Email" button Fixed template email sent, button shows success state
FR-4.5 All emails shall be sent via Brevo SMTP API (server-side only) Must Have All email actions API key never exposed in frontend code

8.2 API Contract

POST Endpoint: doPost(e)

All requests are sent as POST with a JSON body. The action field determines routing.

ActionPayloadResponse
submitLead { name, email, area, price } { success: true, id: "..." } or { success: false, errors: [...] }
updateContacted { id, contacted: true/false, pin } { success: true }
sendEmailToLead { id, pin } { success: true }
verifyPin { pin } { success: true, agentName: "..." }

GET Endpoint: doGet(e)

ParameterResponse
?action=getLeads&pin=XXXX JSON array of all leads + computed stats (totals, percentages, chart data)

11. Risk Assessment

RiskProbabilityImpactMitigation
Google Apps Script rate limits exceeded Low High Quota is 20,000 URL fetch calls/day — sufficient for < 500 leads/day. Monitor usage.
Brevo free tier email limit hit (300/day) Medium Medium Each lead triggers max 3 emails (ack + notify + reminder). Supports ~100 leads/day. Upgrade Brevo plan if exceeded.
Google Sheets performance degrades at scale Low Medium Archive old leads periodically. Sheets supports 10M cells. Consider migration to Firebase at 5,000+ leads.
PIN-based auth is weak security Medium Low Acceptable for v1 single-agent use. Dashboard URL is not indexed by search engines. Upgrade to OAuth in v2.
CORS issues between GitHub Pages and Apps Script Low High Apps Script Web Apps handle CORS natively when deployed as "Anyone can access." Tested during development.
